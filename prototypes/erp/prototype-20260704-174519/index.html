0 · 目标与边界

明确这个模块要解决什么问题，以及哪些数据不能直接下结论。

目标说明最终动作
包裹丢失 识别物流已判定丢失的订单，并判断平台是否已经赔付。 已赔付需索赔接口异常待重试
运费多扣 识别实际运费高于预估运费的订单，并拆分未退货、退货退款两类。 输出多扣金额输出 SKU输出正反向物流金额
核心原则：所有财务金额必须以平台最终结算后的数据为准。未结算订单只能作为预估数据，不进入最终索赔或多扣判断。

1 · 总体处理链路

先识别风险订单，再调用财务接口确认金额，最后生成可执行结果表。

2 · 包裹丢失识别逻辑

先判断包裹是否确实丢失，再进入赔付核查。

2.1 基础筛选

条件最终口径
时间范围最近一个月订单，后续程序可配置为 start_time / end_time。
平台范围Shopee、TikTok Shop。
店铺范围按站点和指定店铺执行，店铺授权需要有效。
订单范围必须存在订单号、店铺 ID、平台、站点、物流单号或包裹信息。

2.2 命中规则

平台接口 / 数据源命中条件
Shopee order/get_order_detail
logistics/get_tracking_info		 package_list.logistics_status = LOGISTICS_LOST
或轨迹中出现 LOST、parcel lost、package lost
或取消原因明确为包裹丢失。
TikTok Shop 订单详情 + 履约物流轨迹 物流状态或轨迹中出现 package lost、parcel lost、lost
或后台订单显示“包裹丢失”。
取消订单不能直接算丢件。必须结合取消原因、物流轨迹、平台丢件状态判断。单纯买家取消、卖家取消、超时取消不进入丢件赔付核查。

3 · 包裹丢失赔付判断逻辑

这里是最终版口径，尤其修正了 TikTok 赔付不在订单维度 statement 中的问题。

3.1 Shopee 赔付判断

项目口径
接口payment/get_escrow_detail
赔付记录来源order_adjustment 或 escrow detail 中的调整记录。
赔付原因识别原因或描述包含 parcel lost、lost compensation、seller_lost_compensation、damaged parcels compensation 等。
金额字段取平台返回的赔付调整金额，币种取 escrow 结算币种。
结论存在赔付记录则 compensation_completed = Y；不存在则 N。

3.2 TikTok Shop 赔付判断

TikTok 不能只调用 /finance/202501/orders/{order_id}/statement_transactions。这个接口只能确认订单自身结算，不一定能看到后续进入其他 statement 的赔付 adjustment。
步骤接口 / 规则说明
1. 拉取 statement GET /finance/202309/statements 按店铺和时间范围拉取 statement，参数必须包含 sort_field=statement_time。
2. 筛选有调整的 statement adjustment_amount != 0 只扫描存在调整金额的 statement，减少调用量。
3. 拉取 statement 明细 GET /finance/202501/statements/{statement_id}/statement_transactions 需要分页拉全，参数使用 sort_field=order_create_time。
4. 匹配赔付交易 type = LOGISTICS_REIMBURSEMENT
adjustment_order_id = 当前订单号		 命中后说明该订单已完成物流赔付。
5. 取金额 adjustment_amount 或 settlement_amount 币种取 statement 的 currency。
TikTok 已赔付判断： 存在 statement_transactions 行： type = LOGISTICS_REIMBURSEMENT adjustment_order_id = order_id 则： compensation_completed = Y actual_compensation_amount = adjustment_amount 或 settlement_amount compensation_reason = LOGISTICS_REIMBURSEMENT

3.3 赔付结果状态

状态判定动作
已完成赔付接口存在明确赔付记录。无需索赔。
未发现赔付记录接口调用成功，但未找到赔付 adjustment。需要发起或跟进索赔。
接口异常接口超时、权限不足、token 失效、返回错误。必须重试，不允许当作未赔付。

4 · 运费多扣识别逻辑

核心是判断物流业者实际收取运费是否高于平台预估运费。

4.1 前置条件：必须已结算

平台结算判断接口最终口径
Shopee payment/get_escrow_list 只有已结算订单，get_escrow_detail 返回的费用才是真实最终费用；未结算订单不进入最终多扣判断。
TikTok Shop Finance statement / transaction 只使用 SETTLED 后的 statement 数据进行判断。

4.2 Shopee 多扣字段

字段含义用途
estimated_shipping_fee预估运费判断基准。
actual_shipping_fee物流业者实际收取运费与预估运费比较。
buyer_paid_shipping_fee买家支付运费计算卖家实际承担影响。
shopee_shipping_rebateShopee 运费回扣 / 补贴抵减卖家承担费用。
seller_paid_shipping_fee_sst卖家支付运费税费计算完整卖家承担金额。
运费多扣命中： actual_shipping_fee > estimated_shipping_fee 运费多扣金额： shipping_overcharge_amount = actual_shipping_fee - estimated_shipping_fee 卖家实际承担影响： seller_shipping_cost = actual_shipping_fee - buyer_paid_shipping_fee - shopee_shipping_rebate + seller_paid_shipping_fee_sst
后台“运费总额”是多个费用项汇总后的展示，不等于单纯多扣金额。判断多扣必须看“物流业者实际收取运费”是否高于“预估运费”。

4.3 多扣原因

原因来源处理方式
接口返回重量 / 尺寸差异记录 listed_weight、actual_weight、listed_dimension、actual_dimension。
接口未返回原因写入备注：后台显示多扣标签，但开放接口未返回完整原因；需后台详情补充。
常见原因实际商品重量 / 尺寸高于刊登商品重量 / 尺寸。

5 · 退货退款场景拆分

运费多扣结果表必须拆成未退货和退货退款两类，避免正反向物流混在一起。

未产生退货

判断条件：

  • 无 return / refund 记录。
  • 无 reverse shipping fee。
  • 订单正常结算。

输出重点：SKU、预估运费、实际运费、买家支付运费、平台补贴、多扣金额。

产生退货退款

判断条件：

  • 存在 return / refund 记录。
  • 或存在 reverse shipping fee / return shipping fee。

输出重点：SKU、正向物流金额、反向物流金额、退款金额、多扣金额。

字段口径
正向物流金额forward_shipping_fee = actual_shipping_fee
反向物流金额reverse_shipping_fee = return_shipping_fee / reverse_shipping_fee
接口无反向物流字段写入“接口未返回反向物流金额”，不允许用 0 代替未知。

7 · 风险点与校验规则

这些规则用于防止误判，也是后续程序必须保留的校验点。

风险点错误做法正确做法
取消订单被误判为丢件 只要订单取消就算丢件。 必须结合物流状态和取消原因，确认是包裹丢失。
TikTok 赔付漏识别 只查订单维度 statement。 按店铺扫描有 adjustment 的 statement，再用 adjustment_order_id 匹配订单。
未结算订单金额不准 未结算也直接取 ES / escrow detail 判断。 必须先用结算接口确认已结算。
接口异常被当成未赔付 接口报错时写 N。 写“接口异常待重试”，不进入最终结论。
反向物流金额缺失 接口没返回就写 0。 写“接口未返回反向物流金额”。

8 · 程序落地建议

后续写成系统功能时，建议按任务批次执行，并保留接口审计。

  1. 先生成批次号 batch_id，所有订单结果和接口原始返回都绑定批次。
  2. 先跑订单和物流识别，生成候选订单；再跑财务核查，避免无效接口调用。
  3. TikTok 赔付核查按“店铺 + statement 时间范围”缓存 statement，避免每个订单重复全量扫描。
  4. 所有接口异常都进入重试队列，不允许写成最终 N。
  5. 结果表固定保留 api_result、api_error、validation_result、remark。
最终结论：包裹丢失的核心是“物流是否确认为丢件 + 财务是否已有赔付”；运费多扣的核心是“已结算后实际运费是否高于预估运费”，并按未退货、退货退款拆分输出。
