0 · 目标与边界

明确这个模块要解决什么问题，以及哪些数据不能直接下结论。

目标说明最终动作
包裹流失 识别物流已判定丢失或损坏无法派送的订单，并判断平台是否已经赔付。 已赔付需索赔接口异常待重试
运费多扣 识别实际运费高于预估运费的订单，并拆分未退货、退货退款两类。 输出多扣金额输出 SKU输出正反向物流金额
核心原则：所有财务金额必须以平台最终结算后的数据为准。未结算订单只能作为预估数据，不进入最终索赔或多扣判断。

3 · 包裹丢失赔付判断逻辑

这里是最终版口径。TikTok 必须通过订单财务接口与结算单交易明细两段链路确认，不允许仅以订单级汇总金额作出待赔付结论。

3.1 Shopee 赔付判断

项目口径
接口payment/get_escrow_detail
赔付记录来源order_adjustment 或 escrow detail 中的调整记录。
赔付原因识别原因或描述包含 parcel lost、lost compensation、seller_lost_compensation、damaged parcels compensation 等。
金额字段取平台返回的赔付调整金额，币种取 escrow 结算币种。
结论存在赔付记录则 compensation_completed = Y；不存在则 N。

3.2 TikTok Shop 赔付判断

TikTok 不能只调用 /finance/202501/orders/{order_id}/statement_transactions，也不能只看“你赚取的金额”汇总。v202501 的订单接口不再直接返回 adjustment_amount 与交易类型；必须先取得 statement_id，再查询结算单交易明细。订单财务接口、结算单明细接口任一失败或未分页完成时，状态只能是“财务待复核”，禁止写成待申请赔付。
步骤接口 / 规则说明
1. 取得结算单号 GET /finance/202501/orders/{order_id}/statement_transactions 从 data.sku_transactions[].statement_id 取该订单关联的所有结算单号并去重。日维度 GET /finance/202309/statements 可用于批处理缓存，但不能替代本步骤。
2. 遍历全部结算单 不以日结算单汇总 adjustment_amount != 0 作为预筛条件 必须扫描步骤 1 返回的每个 statement_id；汇总金额可能无法表达具体订单的赔付交易。
3. 拉取 statement 明细 GET /finance/202501/statements/{statement_id}/statement_transactions 需要分页拉全，参数使用 sort_field=order_create_time。
4. 匹配赔付交易 type = LOGISTICS_REIMBURSEMENT
adjustment_order_id = 当前订单号
兼容 order_id = 当前订单号		 必须遍历 next_page_token 到空。命中正向赔付交易后，说明该订单存在物流赔付入账。
5. 取金额、原因与时间 金额：transactions[].adjustment_amount
原因：transactions[].type
时间：statement.data.create_time
币种：transactions[].currency，回退 statement.data.currency		 settlement_amount 是订单/结算单汇总，不能直接当作赔付金额。时间语义为赔付入账/结算单生成时间，不等同于后台报销流转的创建、审核、成功各节点时间。
TikTok 已赔付判断： 两段接口均调用成功，且所有分页均完成；存在 statement_transactions 行： type = LOGISTICS_REIMBURSEMENT AND (adjustment_order_id = order_id OR order_id = 当前订单号) AND adjustment_amount > 0 则： compensation_completed = Y actual_compensation_amount = SUM(adjustment_amount) compensation_reason = LOGISTICS_REIMBURSEMENT

3.3 赔付结果状态

状态判定动作
已完成赔付接口存在明确赔付记录。无需索赔。
待申请赔付两段接口均成功、全部分页完成，且未找到关联的 LOGISTICS_REIMBURSEMENT。需要发起或跟进索赔。
财务待复核已存在后台结算/入账迹象，但尚未取到交易类型、关联订单或完整分页结果。不直接申请；先补齐结算单交易明细并人工核验。
接口异常接口超时、权限不足、token 失效、返回错误。必须重试，不允许当作未赔付。

3.4 · 自动赔付时效与同步策略

包裹丢失由平台自动发起赔付。系统的目标不是等待到账，而是在平台自动赔付时限内发现未受理案件，预留人工申诉时间，避免超过申诉期限。

统一起算点：T0 = 首次物流明确确认“丢失”或“损坏且无法派送”的事件时间。不能使用订单创建时间、发货时间、取消时间或财务结算时间替代。包裹损坏仍归类为丢件，但保留 lost_sub_type = PACKAGE_DAMAGED。

3.4.1 平台时效与动作节点

平台平台自动赔付自动赔付核查节点人工申诉截止人工动作与预警
Shopee T0 + 7天 T0+1、T0+3、T0+5、T0+6、T0+7 T0 + 15天 T0+8 进入人工申诉队列；T0+12 提醒；T0+14 红色预警。
TikTok Shop T0 + 5天 T0+1、T0+2、T0+3、T0+4、T0+5 T0 + 40天 T0+6 进入人工申诉队列；T0+30 提醒；T0+35 红色预警；T0+39 最后预警。

3.4.2 同步频率

任务建议频率执行口径
已发货、未终态订单物流同步每 2小时发现丢失、损坏无法派送事件后立即建案并固定 T0。
丢件案件赔付/财务核查每 4小时同时核查未结算赔付记录与已结算最终赔付记录。
临近自动赔付或申诉截止每 1小时适用于距自动赔付截止或申诉截止不足 24 小时的案件。
日终全量对账每日 1次补偿接口、订单状态、物流轨迹与案件状态全量回写；仅更新同一案件，不重复建案。

3.4.3 案件状态机

状态进入条件下一动作
DETECTED_LOSS物流已确认丢失或损坏无法派送。建立案件，计算自动赔付截止与申诉截止。
AUTO_COMPENSATION_PENDING已发现赔付调整，但尚未最终结算。继续同步，等待最终结算金额。
AUTO_COMPENSATION_SETTLED已取得最终赔付调整与已结算/已付款财务记录。记录实际赔付金额、币种、时间和原因，关闭案件。
BACKEND_SETTLEMENT_VERIFIEDSeller Center 已明确展示该丢件订单的结算日期和赔付金额，但开放接口尚未返回可关联的赔付交易。按“已完成赔付（后台核验）”关闭索赔动作，同时保留接口补证任务；不得再列为待申请赔付。
AUTO_COMPENSATION_OVERDUE达到平台自动赔付时限，且订单结算单查询、结算单交易明细查询及全部分页均成功，仍未找到匹配的赔付交易。立即进入人工申诉队列，不等待申诉截止日。
FINANCE_REVIEW_REQUIRED后台存在结算/入账迹象，但两段财务接口未能完整证明赔付关联；或任一接口、分页查询异常。进入财务复核和重试队列；禁止标记为待赔付或超时未赔付。
APPEAL_REQUIRED自动赔付超时或人工确认需申诉。输出订单、物流单号、物流证据、应赔金额和截止时间。
APPEAL_DEADLINE_WARNING命中申诉提醒节点。推送运营并升级处理优先级。
API_RETRY_REQUIRED接口异常、权限不足、token 失效或超时。进入重试队列；禁止写成未赔付或超时未赔付。

3.4.4 平台取数口径

平台未结算/处理中最终确认
Shopee订单详情与物流轨迹确认丢件；赔付检查可查询 escrow detail 的调整记录。payment/get_escrow_detail → order_income.order_adjustment[]，取赔付原因、金额、币种、调整时间。
TikTok ShopGET /finance/202507/orders/unsettled 仅作早期预警，不作为最终赔付结论。先调用 GET /finance/202501/orders/{order_id}/statement_transactions 取得 data.sku_transactions[].statement_id；再逐个调用 GET /finance/202501/statements/{statement_id}/statement_transactions 并翻完分页。匹配 type=LOGISTICS_REIMBURSEMENT 且 adjustment_order_id/order_id 等于订单号，实际赔付仅累计 adjustment_amount。
关键限制：财务接口通常提供赔付调整或入账证据，不保证返回 Seller Center 中“已创建报销、处理中”等完整流程历史。因此，达到自动赔付时限且接口未找到赔付时，只能判定为“自动赔付风险/需人工申诉”，不能无限等待财务到账。

3.5 · 赔付校验接口总览、字段与判断

本节是包裹丢失/损坏赔付的开发取数说明。完整返回示例均已脱敏，不包含 access token、refresh token、app secret、签名、店铺 cipher 或买家隐私信息。

3.5.1 接口总览

平台用途接口使用规则
Shopee丢件候选识别GET /api/v2/logistics/get_tracking_info从物流轨迹确认丢失或损坏无法派送，并记录 T0、运单号和物流渠道。
Shopee最终赔付核验GET /api/v2/payment/get_escrow_detail读取 response.order_income.order_adjustment[]；原因命中丢件/损坏赔付时，金额、币种、时间均以该调整记录为准。
TikTok Shop丢件候选识别GET /fulfillment/202309/orders/{order_id}/tracking读取运单号、物流渠道和轨迹，命中 lost、damaged、cannot be delivered 等后建案。
TikTok Shop取得订单关联结算单GET /finance/202501/orders/{order_id}/statement_transactions只用于取得所有 data.sku_transactions[].statement_id；不可用订单 settlement_amount 判断是否赔付。
TikTok Shop最终赔付核验GET /finance/202501/statements/{statement_id}/statement_transactions逐个结算单、逐页查询全部 transactions[]，匹配赔付交易后汇总金额。

3.5.2 请求参数与鉴权边界

平台关键业务参数鉴权/签名要求
Shopeeshop_id、order_sn；物流接口还需订单对应运单信息。partner_id、timestamp、access_token、sign。授权与签名仅在服务端保存，结果表不得输出。
TikTok Shop - 订单结算单路径参数 order_id；查询参数 shop_cipher。app_key、timestamp、sign；请求头 x-tts-access-token。需要财务读取权限。
TikTok Shop - 结算单交易路径参数 statement_id；查询参数 shop_cipher、page_size、page_token、sort_field=order_create_time、sort_order。同上。必须循环到 next_page_token 为空；中断、超时或 429 均不能作未赔付结论。

3.5.3 字段取值逻辑

结果字段Shopee 取值TikTok Shop 取值规则
compensation_completed存在丢件/损坏相关 order_adjustment[]。存在匹配交易且 adjustment_amount > 0。Y；否则不能直接写 N，需先满足全部接口成功和分页完成。
actual_compensation_amountorder_adjustment[].amount 的赔付项金额。所有匹配 transactions[].adjustment_amount 的合计。不得使用 TikTok settlement_amount、订单收入或结算单汇总作为赔付金额。

3.5.4 最终判断逻辑

TikTok Shop： statement_ids = DISTINCT(data.sku_transactions[].statement_id) transactions = 查询每个 statement_id 的全部分页 transactions[] matches = transactions WHERE type = LOGISTICS_REIMBURSEMENT AND (adjustment_order_id = 当前订单号 OR order_id = 当前订单号) AND adjustment_amount > 0 IF 两段接口成功 AND 全部分页完成 AND matches 非空： 已完成赔付；实际赔付金额 = SUM(matches.adjustment_amount) ELSE IF 两段接口成功 AND 全部分页完成 AND matches 为空： 待申请赔付（到达自动赔付时效后转人工申诉） ELSE IF Seller Center 已明确展示结算日期及赔付金额： 已完成赔付（后台核验）；保留接口补证任务 ELSE： 财务待复核 / 接口重试；禁止标为待申请或超时未赔付
状态必要证据后续动作
已完成赔付（接口核验）赔付交易类型、订单关联、金额三项均匹配。关闭索赔；保留 statement_id、adjustment_id 和接口审计记录。
已完成赔付（后台核验）Seller Center 明确展示结算日期和赔付金额，但开放接口未返回关联交易。不再申请；列入接口补证清单。
待申请赔付两段接口成功、全部分页完成，且没有匹配赔付交易。在平台申诉时效内发起或跟进索赔。
财务待复核存在结算迹象，但关联交易、金额或分页证据不完整。先重试和人工核验，不发起重复索赔。
接口重试接口错误、429 限流、权限不足、token 失效或分页中断。重试成功前不改变赔付业务结论。

4 · 运费多扣识别逻辑

核心是判断物流业者实际收取运费是否高于平台预估运费。

4.0 TikTok Shop 两阶段取数口径

TikTok 不能只使用 daily statement 的 shipping_cost_amount 汇总值判断订单级多扣。应先用未结算接口识别实际运费与平台调整，再用已结算 statement 明细确认最终金额。未结算金额仅用于预警，不能直接作为最终申诉/赔付结论。
阶段接口关键字段 / 判断结果口径
未结算预警 GET /finance/202507/orders/unsettled shipping_cost_breakdown.actual_shipping_fee_amount、customer_paid_shipping_fee_amount、shipping_fee_discount_amount、shipping_fee_subsidy_amount；如 type = SHIPPING_FEE_COMPENSATION，表示平台已识别实际运费与预付运费差异。 POTENTIAL_OVERCHARGE 或 COMPENSATION_PENDING，金额为预估值。
已结算最终确认 GET /finance/202501/orders/{order_id}/statement_transactions
GET /finance/202501/statements/{statement_id}/statement_transactions		 先取订单关联的全部 statement_id，再逐页扫描结算单交易；按 adjustment_order_id 或 order_id 关联订单，筛选 SHIPPING_FEE_COMPENSATION；实际赔付仅取并汇总 adjustment_amount，不能使用 settlement_amount 替代。 COMPENSATION_SETTLED，金额为最终值。
TikTok 运费多扣由我方主动申诉。申诉时效暂不写死，必须作为按平台/站点配置的 shipping_overcharge_appeal_days；收到时效后补充预警节点。未出现 SHIPPING_FEE_COMPENSATION 不等于不存在多扣，只表示接口尚未返回平台补偿调整，需要保留为待申诉/待核对案件。

4.1 前置条件：必须已结算

平台结算判断接口最终口径
Shopee payment/get_escrow_list 只有已结算订单，get_escrow_detail 返回的费用才是真实最终费用；未结算订单不进入最终多扣判断。
TikTok Shop Finance statement / transaction 只使用 SETTLED 后的 statement 数据进行判断。

4.2 Shopee 多扣字段

字段含义用途
estimated_shipping_fee预估运费判断基准。
actual_shipping_fee物流业者实际收取运费与预估运费比较。
buyer_paid_shipping_fee买家支付运费计算卖家实际承担影响。
shopee_shipping_rebateShopee 运费回扣 / 补贴抵减卖家承担费用。
seller_paid_shipping_fee_sst卖家支付运费税费计算完整卖家承担金额。
运费多扣命中： actual_shipping_fee > estimated_shipping_fee 运费多扣金额： shipping_overcharge_amount = actual_shipping_fee - estimated_shipping_fee 卖家实际承担影响： seller_shipping_cost = actual_shipping_fee - buyer_paid_shipping_fee - shopee_shipping_rebate + seller_paid_shipping_fee_sst
后台“运费总额”是多个费用项汇总后的展示，不等于单纯多扣金额。判断多扣必须看“物流业者实际收取运费”是否高于“预估运费”。

4.3 多扣原因

原因来源处理方式
接口返回重量 / 尺寸差异记录 listed_weight、actual_weight、listed_dimension、actual_dimension。
接口未返回原因写入备注：后台显示多扣标签，但开放接口未返回完整原因；需后台详情补充。
常见原因实际商品重量 / 尺寸高于刊登商品重量 / 尺寸。

5 · 退货退款场景拆分

运费多扣结果表必须拆成未退货和退货退款两类，避免正反向物流混在一起。

未产生退货

判断条件：

  • 无 return / refund 记录。
  • 无 reverse shipping fee。
  • 订单正常结算。

输出重点：SKU、预估运费、实际运费、买家支付运费、平台补贴、多扣金额。

产生退货退款

判断条件：

  • 存在 return / refund 记录。
  • 或存在 reverse shipping fee / return shipping fee。

输出重点：SKU、正向物流金额、反向物流金额、退款金额、多扣金额。

字段口径
正向物流金额forward_shipping_fee = actual_shipping_fee
反向物流金额reverse_shipping_fee = return_shipping_fee / reverse_shipping_fee
接口无反向物流字段写入“接口未返回反向物流金额”，不允许用 0 代替未知。

7 · 风险点与校验规则

这些规则用于防止误判，也是后续程序必须保留的校验点。

风险点错误做法正确做法
取消订单被误判为丢件 只要订单取消就算丢件。 必须结合物流状态和取消原因，确认是包裹丢失。
包裹损坏漏归类 只识别 lost，不识别 damaged。 损坏且无法派送也归入“丢件”，二级类型标记为“包裹损坏”。
TikTok 赔付漏识别 仅按订单结算金额或单一 statement 判断。 先按订单取得全部 statement_id，再扫描每个结算单的全部分页交易；以 LOGISTICS_REIMBURSEMENT 和 adjustment_order_id/order_id 双重关联确认。
未结算订单金额不准 未结算也直接取 ES / escrow detail 判断。 必须先用结算接口确认已结算。
接口异常被当成未赔付 接口报错时写 N。 写“接口异常待重试”，不进入最终结论。
反向物流金额缺失 接口没返回就写 0。 写“接口未返回反向物流金额”。

8 · 程序落地建议

后续写成系统功能时，建议按任务批次执行，并保留接口审计。

  1. 先生成批次号 batch_id，所有订单结果和接口原始返回都绑定批次。
  2. 先跑订单和物流识别，生成候选订单；再跑财务核查，避免无效接口调用。
  3. 包裹流失识别需要同时输出 lost_category 和 lost_sub_type，便于区分物流丢失和包裹损坏。
  4. TikTok 赔付核查缓存“订单号 → statement_id 列表”及结算单分页交易；不得以订单 settlement_amount 直接判断赔付。
  5. 所有接口异常都进入重试队列，不允许写成最终 N。
  6. 结果表固定保留 api_result、api_error、validation_result、remark。
最终结论：包裹流失的核心是“物流是否确认为丢失或损坏无法派送 + 财务是否已有赔付”；运费多扣的核心是“已结算后实际运费是否高于预估运费”，并按未退货、退货退款拆分输出，同时展示物流单号、物流渠道和店铺名称。
